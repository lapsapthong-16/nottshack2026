# EvoGuard: Implementation Plan (High-Fidelity Log Edition)

**Goal:** Transform the Dash Platform from a simple result registry into a transparent audit archive by storing full AI analysis logs alongside risk scores.

---

## Phase 1: Data Contract Evolution (v2)
**Goal:** Modify the `auditReport` schema to accommodate detailed log data while respecting Platform limits.

1.  **Schema Update:**
    *   Add a `logs` property to the `auditReport` document.
    *   **Type:** `string` (or `byteArray` for optimized storage).
    *   **Constraint:** Set a `maxLength` at ~15,000 characters to stay within the 16KB document limit.
2.  **Version Management:**
    *   Increment the contract version in `lib/server/schema/evoguardContract.ts`.
    *   Re-deploy or update the contract on Testnet.
3.  **Local Sync:** Update `.env` with the new `CONTRACT_ID` if a fresh deployment is made.

---

## Phase 2: The Storage Pipeline (Compression & Logic)
**Goal:** Ensure logs are stored efficiently and reliably.

1.  **Compression Logic:** 
    *   Implement server-side compression (using Node's `zlib`) to pack large `LOGS.md` content before sending to Dash.
    *   This allows ~50-100KB of raw text to fit into the 16KB document limit.
2.  **Storage API Update (`/api/evoguard/document/store`):**
    *   Modify the handler to accept an optional `logs` parameter.
    *   Read the actual `LOGS.md` generated by the AI scanner and attach it to the `auditReport` payload.
3.  **Truncation Safety:** Add a safety check that trims the end of the logs if the payload still exceeds 16KB after compression.

---

## Phase 3: High-Fidelity UI (The Dash Explorer)
**Goal:** Provide a "Glass Box" experience for developers to inspect the AI's reasoning.

1.  **Reports Table (`reports.tsx` / `report.tsx`):**
    *   Update the "View Details" expansion to fetch and display the `logs` field.
    *   Implement an "Audit Log Viewer" component with syntax highlighting (Markdown/JSON) for the logs.
2.  **Verification Badge:**
    *   Show "Verified Log" only if the document's `auditorSignature` matches the `evoguard.dash` identity.
3.  **Explorer Deep Link:** Maintain the link to the official Dash Platform Explorer for raw document verification.

---

## Phase 4: CLI Integration
**Goal:** Allow developers to fetch full logs locally before installing packages.

1.  **`evoguard fetch-logs <pkg>@<ver>`:**
    *   Query the Dash Platform for the latest report.
    *   Download and decompress the logs.
    *   Output the logs to the terminal or a local file for the developer to review.

---

## Success Criteria (Phase 2 & 3 Combined)
*   **Action:** Audit `express@latest`, scan generates a 12KB `LOGS.md`.
*   **Dash Drive:** Report is stored with `riskScore`, `summary`, and the full `logs`.
*   **UI:** User clicks a report and can read exactly *why* the AI flagged `allowPrototypes: true` in `utils.js`.
